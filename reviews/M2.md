## Findings

### 1. [required — correctness, cross-platform] Venv escape via symlink resolution in `_start_child`

`src/gauntlet/adapters/python_callable.py:283`:

```python
str(Path(sys.executable).resolve()),
```

On POSIX, a virtualenv's `bin/python` is a **symlink** to the base interpreter. `.resolve()` follows it, so the child is spawned with (e.g.) `/usr/bin/python3.11` instead of the venv interpreter. That interpreter has no `gauntlet` installed, the worker dies at import with `ModuleNotFoundError` (exit code 1), and every subprocess-dependent test fails — exactly the 13 observed failures (all of `test_python_callable_adapter.py`'s spawn tests plus the two `test_sample_agent.py` subprocess tests). On Windows, venv `python.exe` is a real file, so `.resolve()` is a no-op — which is why Codex's gate honestly passed.

**Required fix:** spawn with `sys.executable` unchanged (it is already an absolute path):

```python
command = [
    sys.executable,
    "-P",
    "-s",
    ...
```

**Reviewer verification of the fix:** applying exactly this one-line change to a scratch copy of `5d1b9f5` and re-running the full suite on Linux yields **107 passed**. No other change is needed. (The scratch patch was reverted; the fix must land as a Codex commit.)

**Required follow-up with the fix commit:** re-run the full M2 gate on Windows as well and record both results in `HANDOFF.md`, confirming the change regresses nothing there (none expected — on Windows the two expressions produce equivalent spawns). Commit as `WP-2.2 fix: <summary>`.
