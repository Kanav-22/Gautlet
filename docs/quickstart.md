# GAUNTLET quickstart

This path is designed to reach a first local, evidence-backed evaluation within
the project's fifteen-minute developer-experience goal. That is a design target,
not a measured performance claim.

## 1. Install from this checkout

GAUNTLET requires Python 3.11 or newer. From the repository root:

```bash
python -m venv .venv
```

Activate the environment:

```powershell
.\.venv\Scripts\Activate.ps1
```

On Linux or macOS, use `source .venv/bin/activate` instead. Then install and
verify the CLI:

```bash
python -m pip install -e ".[dev]"
gauntlet --version
```

## 2. Check the local environment

Choose a writable artifact directory explicitly so the same command works in
restricted development environments:

```bash
gauntlet doctor --artifact-root .gauntlet/demo-artifacts
```

`doctor` is fully offline. It checks the Python version, required imports,
subprocess startup, atomic artifact writes, and the built-in policy and
benchmark pack.

## 3. Validate and inspect

```bash
gauntlet benchmark validate benchmarks/agent_mvp
gauntlet inspect examples
```

The benchmark command should report `gauntlet.agent.mvp` version `0.1.0`
with 15 scenarios. Inspection is static: it parses Python source without
importing or executing the sample project.

## 4. Run the flagship evaluation

The sample configuration is committed at `examples/.gauntlet/config.yaml`.
Run the complete pack three times with a fixed seed:

```bash
gauntlet evaluate examples --benchmark gauntlet.agent.mvp --seed 42 --repeat 3 --offline --artifact-root .gauntlet/demo-artifacts --verbose
```

The command is non-interactive and prints the run ID, score, release
recommendation, reproducibility result, and exact Markdown report path. A
successful deterministic sample run exits 0 and writes:

```text
.gauntlet/demo-artifacts/
└── runs/
    └── run_<timestamp>_<id>/
        ├── manifest.json
        ├── environment.json
        ├── config.resolved.yaml
        ├── results.json
        ├── scorecard.json
        ├── findings.json
        ├── canonical.json
        ├── report.md
        └── evidence/
```

The checked-in example at
`examples/reports/agent_mvp/report.md` was generated by this real sample
evaluation and is intentionally limited to redacted summary data.

Run identity, timestamps, latency, and raw evidence hashes can vary. ADR-004
repeatability applies to `canonical.json`, which excludes documented
lifecycle fields while retaining output, assertion, metric, and redacted
evidence semantics. Live or network-enabled runs report observed repeat
distributions and never claim determinism.

## 5. Compare runs

List run IDs and compare a candidate against an earlier baseline:

```bash
gauntlet runs list --artifact-root .gauntlet/demo-artifacts
gauntlet compare BASELINE_RUN_ID CANDIDATE_RUN_ID --artifact-root .gauntlet/demo-artifacts
```

`compare` reports score changes, fixed and new failures, latency and
adapter-reported cost changes, plus configuration, benchmark, and environment
context. A proven deterministic regression exits 1. A configuration change is
reported as a context change rather than invented as a regression.

## Exit codes

| Code | Meaning |
|---:|---|
| 0 | Evaluation completed and release policy passed |
| 1 | Evaluation completed and release policy failed |
| 2 | Configuration or input error |
| 3 | Execution error |
| 4 | Security boundary violation |
| 5 | Incomplete or insufficient evaluation |

## Isolation boundary

`--offline` forces fixture-mode network policy and removes network credentials
and proxy variables from the child environment. It does **not** provide
operating-system-level socket denial. MVP subprocess isolation provides process
separation, parent-enforced deadlines, bounded protocol messages, and injected
stub tools; it is not a hardened sandbox for hostile code. Evaluate only code
you trust to run on the local machine.
